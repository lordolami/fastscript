# FastScript Agent Positioning

## Purpose

The FastScript coding agent exists to make FastScript the clean internal reasoning language for software generation, repair, migration, deployment, and export.

It is not just an assistant.

It is intended to become:

1. a software builder
2. a full-stack coding agent
3. a migration engine
4. a deploy agent
5. a design generator
6. a multi-target exporter
7. eventually, a general coding model platform grounded in FastScript

## Core Thesis

Most coding models reason directly in ecosystem chaos:

1. raw JavaScript
2. TypeScript
3. framework-specific conventions
4. inconsistent project structures
5. multiple conflicting patterns for the same task

The FastScript agent should reason differently:

1. think in FastScript first
2. keep FastScript as the canonical authored language
3. validate against the FastScript compiler and runtime
4. export outward to other targets when needed

This gives:

1. smaller reasoning surface
2. lower ambiguity
3. higher validation quality
4. better generation speed
5. better training data quality
6. stronger product reliability

## Why It Is Different

### 1. FastScript is the internal coding substrate

The agent should not merely support FastScript as one output among many.

It should use FastScript as:

1. internal planning language
2. canonical authored language
3. validation target
4. export source

### 2. Compiler-loop grounded generation

The FastScript agent should operate through a proof loop:

1. generate FastScript
2. run build
3. run validate
4. inspect diagnostics
5. patch errors
6. export or deploy only after proof passes

### 3. Same-host full-stack default

The FastScript agent should think in one-app boundaries by default:

1. pages
2. APIs
3. middleware
4. auth
5. data
6. jobs
7. deploy target

This should happen without a forced frontend/backend mental split.

### 4. Export without losing generation speed

The model should not need to think in every target language first.

Instead:

1. generate FastScript quickly
2. use exporters/adapters for target ecosystems
3. keep FastScript speed while reaching broader deployment targets

### 5. Lower-cost model viability

FastScript is intentionally smaller and more regular than raw multi-framework web stacks.

That means:

1. weaker models can still be useful
2. token usage can be lower
3. inference can be faster
4. reliability can be higher at lower cost

## Product Categories

The FastScript agent should eventually support both:

### Builder mode

For users who want:

1. prompt to website
2. prompt to dashboard
3. prompt to ecommerce app
4. prompt to full-stack product
5. prompt to redesign or extend a live app

### Developer mode

For users who want:

1. codebase editing
2. migration
3. repair
4. review
5. deployment
6. export
7. repo-scale coding workflows

## Long-Term Product Promise

The long-term promise is:

1. non-developers can describe software and get real deployable products
2. developers can build faster than with existing coding agents
3. teams keep full ownership of their source
4. FastScript remains the clean internal system
5. exports remain available when teams need other targets

## Non-Negotiable Principles

1. FastScript remains the canonical authored language
2. the agent must be compiler-loop grounded
3. same-host full-stack is the default mental model
4. exports are extensions, not replacements for the core language
5. the system must remain usable by both humans and weaker models
6. speed and regularity are part of the moat, not side effects
